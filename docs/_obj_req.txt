SHAKTI
Supply Chain Management System Product Requirements
0. Introduction
0.1 Project Overview
This note describes users, tasks, and journeys for a supply chain management system, named Shakti. Its purpose is to reduce the amount of manual work done by users and streamlining the movement and visibility of inventory.
•        Real time visibility of the location of each item of inventory
•        MIS reporting
•        Reducing time taken for stock taking, movement of inventory
•        Reducing cost
0.2 Platform & Device Requirements
The system is being designed as a web application. However, most of the users will be operating from a mobile. It should be usable across Android and iOS mobiles, and Windows or MacOS desktops/laptops.
The system should be accessible from Chrome, Safari, Bing. It should be accessible on a device running MacOS, Windows, or Linux. 
0.3 Branding & Typography
•        The brand colours to be used are Purple and Orange. Grey can be used for highlighting these colours.
•        The font to be used through the application is Raleway Light.
0.4 Responsive Design Requirements
Since most users will operate from mobile devices, the application must be responsive across screen sizes. Specific requirements:
•        Breakpoints at 768px (tablet) and 480px (mobile).
•        Two column grid layouts must stack to single column at 640px.
•        Filter tabs must wrap gracefully on narrow screens.
•        Form input font size must be at least 16px to prevent iOS Safari auto zoom on focus.
•        Data tables must scroll horizontally inside their container cards with a minimum width of 480px to remain readable.
•        Navigation and main content areas must adapt to mobile viewport widths.
0.5 Global System Rules
0.5.1 Authentication
The system when accessed through the web presents to all users a user ID and password login console. A user can access their account through an innoviti domain login ID and password through Google SSO only (For Now exclude the Google SSO authentication)
The system should support up to 30 concurrent users without any performance degradation.
 
0.5.2 API First Architecture
All features described in this document must be implemented as APIs first. Every capability object creation, data loading, auditing, order management, dispatch, reporting must be exposed as a REST API endpoint. The UI described in the journeys below is a layer built on top of these APIs. This ensures that any operation possible through the UI is equally possible programmatically.
Any object update that is possible through a user interface should also be possible through an API
 
0.5.4 Data Backup
All object data must be backed up with timestamps, and this backup should not be directly accessible by the main application and should be usable in case of a system failure.

1.1.2 User Fields
Every User has the following fields defined:
•        User Index (auto incremented starting with UIN (5 digits, starting with 10001). User index should be auto incremented on creation of a new user.
•        First name (compulsory)
•        Last name (compulsory)
•        User Type (picklist from User Types)
•        Email address (unique email)
•        Mobile number (compulsory, Indian phone number format)
•        Vendor associated with (for all user types other than Repair Line Users and Logistics Users, the default Vendor is Innoviti).
•        Employee ID if Vendor is Innoviti(The format is IC/NNNN and INN/NNNN;(IC/ and INN/ followed by 4 digits); example:IC/0000)
•        Address Line 1: Address line 1 for receiving shipments
•        Address Line 2: Address line 2 for shipments
•        City (derived from Pin Code)
•        State (derived from Pin Code)
•        Pincode
•        Status: Picklist with values Active, Inactive. Defaults to Active on creation. It is set to Inactive if a user leaves. The user profile is maintained for audit purposes. Can be changed by Admin or Super Admin only.
Inactive users are skipped during order generation, retrieval generation, and other automated flows.

1.4 Stock Keeping Unit (SKU)
SKU is used to track different types of items used in the process of assembly. Two SKUs are different if their specifications are different.
Examples:
•        Consumables (like brown tape, packaging material), payment assets (like terminal)
•        Tools (like pliers, tweezers etc.), accessories (like cables, adaptors) that have a short life but can be used multiple times
•        Assembly line assets (test setups, computers, etc.).
•        Spare parts that are used for repairing terminals
•        Scrap payment assets (terminals that are being scrapped).
1.4.1 SKU Types
SKU types are not hardcoded property; The admin can create new SKU type in future if needed, for now these are the SKU types
•        Payment Terminal
•        Base Station
•        SIM Card
•        Assembly Line Assets
•        Adaptors
•        USB cables
•        Paper rolls
•        Tools
•        Consumables
•        Spare Parts
 
1.4.2 SKU Fields
Every SKU has the following fields defined:
•        SKU number   starting with INN (5 digit number starting with 10001).
•        SKU name – this is for easier recall of the SKU.
•        Description – a text box to describe the part.
•        SKU Tracking Method (STM) – This is used for tracking a unit of an SKU and in reporting. This can be of 2 types:
o   Serial: If each unit of an SKU needs to be tracked independently then a unique serial number needs to be associated with each unit. Example payment terminals, base station, SIM cards.
o   None: If units of an SKU are indistinguishable. Example cables.
•        SKU Type (picklist from SKU Types)
•        In case the SKU Type is a Payment Terminal, then the following two additional fields need to be selected:
o   Associated Adaptor SKU Name (compulsory): The adaptor to be used with this SKU (from Adaptor picklist). This can be multi select, and all the adaptors selected will show.
o   Associated USB Cable Name (compulsory): The USB cable to be used with this SKU (from USB cable picklist). This can be multi select and all the adaptors selected will show.
•        Specifications – PDF document prepared by Innoviti with all details of the SKU as required to order.
•        Approximate price per unit (MOQ, unit price)
•        Status: Active, Inactive (for old SKUs that are no longer in use).
•        SKU Vendor (picklist from SKU Vendor list) ….Mandatory
 
1.4.3 Vendor SKUs and their association with SKUs
A Vendor SKU is a vendor's own product, defined once independently of any Innoviti SKU. A Vendor SKU can supply more than one Innoviti SKU, and an Innoviti SKU can have more than one Vendor SKU associated with it — the relationship is many-to-many, managed from the SKU definition. Each Vendor SKU has the following fields:
•        Vendor SKU number – the part number that the vendor uses; unique within that vendor.
•        Vendor SKU name – the vendor's name for the part.
•        Vendor SKU specification – PDF provided by the vendor for that part.
•        Vendor SKU price (MOQ, unit price).
•        Status: Active, Inactive.
For each Innoviti SKU, one of its associated Vendor SKUs may be marked as the default supplier.
 
1.5 Inventory Locations
At any point of time every inventory item has to be in the location of a vendor. During transit the state and not the location of the asset changes, the location changing only after it is received at the destination. Each location has to be tagged to a vendor, and each location has a principal contact and secondary contact defined.
The following are the fields for every Location:
•        Location Index (auto incremented, starting with LIN (8 digits, starting with 10000001).
•        Vendor tagged to: From vendor picklist
•        Location Name: Entry
•        Location Address:
o   Address Line 1
o   Address Line 2
o   Pin Code
o   City (derived from Pin Code)
o   State (derived from Pin Code)
•        Principal Contact Person 1 at Location (Can be picked from Contacts registered against that Vendor)
•        Secondary Contact Person 2 at Location (Can be picked from Contacts registered against that Vendor)
